# Решение задания 2: Основы Машинного Обучения

## Часть 1: Ответы на теоретические вопросы

### Вопрос 1
**Обучение с учителем** использует размеченные данные (с правильными ответами). Алгоритм учится сопоставлять входы с известными выходами.
- Пример: Классификация emails на спам/не спам

**Обучение без учителя** работает с неразмеченными данными. Алгоритм сам ищет структуры и закономерности.
- Пример: Сегментация клиентов по покупкам

### Вопрос 2
**Классификация** предсказывает категорию/класс объекта (дискретное значение).
- Пример: Кошка или собака, спам или не спам

**Регрессия** предсказывает непрерывное числовое значение.
- Пример: Цена дома, температура, рост

**Предсказание цены акции** — это задача регрессии, так как цена является непрерывным числовым значением.

### Вопрос 3
**Переобучение** — когда модель слишком хорошо запомнила обучающие данные (включая шум) и плохо обобщает на новые данные.

**Три способа борьбы:**
1. Регуляризация (L1, L2, Dropout)
2. Увеличение количества обучающих данных
3. Упрощение модели (меньше параметров)
4. Кросс-валидация
5. Early Stopping (ранняя остановка)

### Вопрос 4
**Разделение данных нужно для:**

- **Обучающая выборка (Train)**: На этих данных модель обучается, подбирает веса
- **Валидационная выборка (Validation)**: Для настройки гиперпараметров и выбора лучшей модели
- **Тестовая выборка (Test)**: Для финальной оценки качества на совершенно новых данных

Это позволяет объективно оценить, насколько модель способна обобщать, а не просто запоминать.

### Вопрос 5
**Как работает KNN:**
1. Для нового объекта находим K ближайших соседей в обучающей выборке
2. Смотрим классы этих соседей
3. Присваиваем объект к классу, который встречается чаще всего (мажоритарное голосование)

**При слишком маленьком K (например, K=1):**
- Модель чувствительна к шуму и выбросам
- Высокая дисперсия, возможно переобучение

**При слишком большом K:**
- Модель становится слишком гладкой
- Может игнорировать локальные паттерны
- Высокое смещение, возможно недообучение

### Задание 2.2: Вычисление метрик

**Дано:**
| | Предсказано СПАМ | Предсказано НЕ СПАМ |
|--|------------------|---------------------|
| **Фактически СПАМ** | TP = 80 | FN = 20 |
| **Фактически НЕ СПАМ** | FP = 10 | TN = 890 |

**Вычисления:**

1. **Accuracy** = (TP + TN) / (TP + TN + FP + FN)
   = (80 + 890) / (80 + 890 + 10 + 20)
   = 970 / 1000
   = **0.97 (97%)**

2. **Precision** = TP / (TP + FP)
   = 80 / (80 + 10)
   = 80 / 90
   = **0.889 (88.9%)**

3. **Recall** = TP / (TP + FN)
   = 80 / (80 + 20)
   = 80 / 100
   = **0.80 (80%)**

4. **F1-Score** = 2 × (Precision × Recall) / (Precision + Recall)
   = 2 × (0.889 × 0.80) / (0.889 + 0.80)
   = 2 × 0.7112 / 1.689
   = 1.4224 / 1.689
   = **0.842 (84.2%)**

**Анализ:**

**Хорошая ли это модель?**
Да, это хорошая модель:
- Accuracy 97% — высокий общий процент правильных ответов
- Precision 89% — когда модель говорит "спам", в 89% случаев это действительно спам
- Recall 80% — модель находит 80% всего спама

**Какую метрику важнее максимизировать?**

Для фильтра спама важнее **Precision**, потому что:
- Ложноположительный результат (FP) — важное письмо попало в спам — это серьёзная проблема
- Ложноотрицательный результат (FN) — спам попал во входящие — менее критично, пользователь сам удалит

Однако нужен баланс: слишком высокий Precision при низком Recall означает, что мы пропускаем много спама.

## Часть 4: Пример решения кейс-стади

### План проекта: Оценка кредитоспособности

#### 1. Данные

**Какие данные понадобятся:**
- Демографические: возраст, пол, образование, семейное положение
- Финансовые: доход, расходы, кредитная история, наличие активов
- Поведенческие: история платежей, количество просрочек
- Кредитные: сумма кредита, цель, срок

**Где взять:**
- Внутренняя база данных банка (исторические заявки)
- Кредитные бюро (external credit scores)
- Заполненные заёмщиком анкеты

**Проблемы:**
- Пропущенные значения (не все заполняют анкету полностью)
- Дисбаланс классов (дефолтников меньше, чем платёжных клиентов)
- Устаревшие данные (экономическая ситуация меняется)

#### 2. Выбор модели

**Тип обучения:** Обучение с учителем, бинарная классификация (дефолт/нет дефолта)

**Алгоритмы:**
- **Логистическая регрессия**: Базовый интерпретируемый метод
- **Random Forest**: Хорошая точность, устойчивость к выбросам
- **Gradient Boosting (XGBoost/LightGBM)**: State-of-the-art для табличных данных
- **Нейронные сети**: Если очень много данных

**Почему:** Ансамблевые методы (RF, GB) обычно показывают лучшую точность на табличных данных и устойчивы к переобучению.

#### 3. Метрики

**Как оценивать:**
- **ROC-AUC**: Общая способность различать классы
- **Precision-Recall曲线**: Важно при дисбалансе классов
- **Confusion Matrix**: Для понимания типов ошибок

**Наиболее важная метрика:**
Для банка важнее минимизировать **False Positive** (выдать кредит неплатёжному клиенту), поэтому:
- **Precision для класса "дефолт"** — сколько из предсказанных дефолтников действительно не вернут
- Но также важен **Recall** — не упустить слишком много плохих клиентов

**Баланс:** Зависит от риск-аппетита банка. Консервативный банк предпочтёт высокий Precision.

#### 4. Риски

**Этические вопросы:**
- **Дискриминация**: Модель может дискриминировать по полу, возрасту, расе
- **"Чёрный ящик"**: Сложные модели трудно интерпретировать
- **Self-fulfilling prophecy**: Отказ в кредите ухудшает финансовое положение

**Как избежать дискриминации:**
- Исключить защищённые признаки (пол, раса, возраст) из модели
- Проверять модель на разных демографических группах
- Использовать fairness-метрики
- Регулярный аудит модели

**Ошибочные предсказания:**
- Дать возможность апелляции
- Человеческий обзор пограничных случаев
- Постоянный мониторинг и переобучение

#### 5. Внедрение

**Интеграция:**
- API для скоринга в реальном времени
- Интеграция с CRM и системой принятия решений
- A/B тестирование перед полным запуском

**Финальное решение:**
- Модель даёт рекомендацию (score)
- Человек (кредитный офицер) принимает финальное решение для пограничных случаев
- Автоматическое одобрение/отказ только для очевидных случаев

**Мониторинг:**
- Track drift данных (распределение входных данных меняется?)
- Track performance (точность падает со временем?)
- Регулярное переобучение (раз в квартал/год)
- Alert система при аномалиях
